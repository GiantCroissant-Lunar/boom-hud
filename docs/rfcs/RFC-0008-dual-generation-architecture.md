# RFC-0008: Dual Generation Architecture (Design→UI + MVVM Adapters)

- **Status**: Accepted
- **Created**: 2025-12-12
- **Authors**: BoomHud Contributors

## Summary

Adopt a dual-generation approach:

- **Layer A (Design → IR → UI)**: a CLI / .NET tool that consumes Figma JSON (and optional token sources) and generates framework UI artifacts.
- **Layer B (MVVM Binding Adapters)**: a Roslyn incremental source generator that adapts the MVVM surface to different MVVM frameworks (ReactiveUI, CommunityToolkit, custom patterns) while keeping the generated views MVVM-framework-agnostic.

## Motivation

BoomHud needs:

- Multi-file output and reusable component generation (Figma COMPONENT/INSTANCE)
- Runtime theme switching and token-driven styling
- MVVM integration that is flexible across frameworks

A single generator cannot easily satisfy both constraints:

- CLI generation is better for multi-file output and complex IO.
- Roslyn generation is better for project-specific MVVM wiring and compile-time integration.

## Goals

- Define responsibilities and boundaries between Layer A and Layer B
- Define the stable contract between them
- Define how the CLI integrates into MSBuild as an optional build step
- Define how MVVM frameworks are selected/configured

## Non-Goals

- Replace the CLI with a full Roslyn generator for UI file emission
- Plugin loading for third-party backends (in-repo backends only)

## Design

### Layer A: CLI / .NET Tool

#### Inputs

- Figma JSON export (and/or later: BoomHud DSL)
- Optional token sources:
  - Figma Variables JSON
  - Design tokens JSON

#### Outputs

- One or more generated files per backend:
  - Terminal.Gui: `.g.cs`
  - Avalonia: `.axaml`, `.axaml.cs`, `.g.cs` (VM interface)
- Theme resources per variant
- Extracted reusable components (when Figma COMPONENT/INSTANCE is detected)

#### Invocation & MSBuild Integration

Initially, the CLI is **manual-only** (invoked explicitly by the developer or tooling).

MSBuild integration (running the CLI automatically before compile) is explicitly deferred to a later RFC/update once the generator outputs and incremental behavior stabilize.

When MSBuild integration is introduced, the generated output should go under an intermediate directory (e.g. `obj/Generated/BoomHud/...`) and be included as `Compile` / `AvaloniaXaml` items.

### Layer B: MVVM Binding Adapter Generator (Roslyn)

Layer B consumes the stable contract generated by Layer A and emits additional sources in the consuming project.

#### Stable Contract

- Generated **interfaces** `I{ViewName}ViewModel`
- Generated metadata describing:
  - required properties
  - required commands
  - expected types (where available)

Layer A must not emit hard references to MVVM libraries.

#### MVVM Flavors

Layer B is configured via MSBuild properties (similar to UnifyEcs):

- `BoomHudMvvmFlavor = None | ReactiveUI | CommunityToolkit | Custom`
- Flavor-specific additional properties may be provided.

### Diagnostics

- Layer A emits diagnostics via CLI output and/or a report file.
- Layer B emits diagnostics via Roslyn diagnostics.

## Backward Compatibility

- Existing CLI behavior remains valid.
- Roslyn generator is optional and additive.

## Alternatives Considered

- Pure Roslyn UI generator (rejected: awkward for multi-file non-C# outputs and external file IO)
- Pure CLI MVVM integration (rejected: cannot adapt to per-project MVVM framework choices as cleanly)

## Open Questions

- Should Layer A emit additional strongly typed metadata (e.g. JSON schema or C# records) to make Layer B simpler?
- Should MVVM flavor selection live in the CLI, in Roslyn, or both?
